## 二、架构总览

### 2.1 后端分层

```
FastAPI (main.py)
├── 中间件栈 (LIFO)：Prometheus → RateLimit → Logging
├── api/         16 个路由模块（auth/sessions/options/orders/...）
├── services/    8 个业务域（dispatch/order/payment/logistics/hunyuan3d/llm/tasks/workorder/alerting）
├── core/        9 个基础设施（config/crypto/redis/storage/rate_limit/metrics/time/prompt_defense/logging）
├── models/      entities(14表) + types(Vector/EncryptedStr/JSONB)
└── db/          async session + init_db
```

分层清晰，service 与 api 解耦良好，core 沉淀了横切关注点。这是本项目工程质量的最大亮点。

### 2.2 前端分层

```
Vue 3 + TS + Vite
├── views/       用户端(5) + studio/(3) + admin/(4)
├── components/   8 个（ChatPanel/OptionCards/PreviewCanvas/ThreeViewer/...）
├── composables/  3 个（useHunyuan3D/useDesignVersions/usePreviewRotation）
├── stores/       auth（Pinia setup 风格）
├── api/          client(axios 拦截器) + modules(端点封装)
└── constants/types  glaze / design 类型
```

DesignView 采用三栏组合式布局，组件拆分合理，composables 抽离了可复用逻辑。

### 2.3 数据流关键链路

```
对话设计：用户输入 → LLM parser → design_params → options 生成 → DesignVersion(多版本)
3D 生成：  Hunyuan3D submit → task 双写(Redis+PG) → worker 轮询 → GLB → MinIO
派单：     order 创建 → 四维评分(scoring) → policy 阈值 → dispatch → 工作室
履约：     studio 接单 → 状态机流转 → 物流 provider → 支付回调(幂等)
实时：     SSE + Last-Event-ID 断点续传
```

## 三、代码质量分析

### 3.1 后端代码质量 ⭐⭐⭐⭐☆ (4/5)

**优秀实践**

1. **配置管理规范**（`config.py`）：Pydantic Settings + 环境变量；生产环境启动强校验，拒启占位密钥（`_DEV_PLACEHOLDERS`）；类型安全 `Literal["development","staging","production"]`。
2. **加密实现健壮**（`crypto.py`）：AES-GCM 字段透明加密（`EncryptedStr`）；HMAC-SHA256 手机号哈希 + pepper；PBKDF2 密钥派生。已修复 pepper 不一致导致个人资料空白问题。
3. **中间件设计考究**（`main.py`）：LIFO 顺序精确控制，Prometheus 在最外层覆盖 RateLimit 的 429，RateLimit 在 Logging 外层避免 429 噪音；structlog JSON + contextvars 请求 ID 自动传递。
4. **API 设计**：RESTful 规范；幂等性保证（`IdempotencyKey` 表 + 唯一约束）；全局异常 handler 不泄露 traceback。
5. **ORM 使用**：SQLAlchemy 2.0 异步风格一致；关系定义清晰；索引到位（`phone_hash`、`idempotency_key`、`expires_at`）。

**待改进**

1. **类型标注不完整**：部分函数缺返回类型（如 `dispatcher.py` `dispatch_to_studio`）。建议启用 mypy 严格模式。
2. **测试覆盖不均**：130 个测试集中在 auth/crypto/rate_limit/metrics，缺 dispatch 评分、order 状态机、hunyuan3d worker。目标 70%+。
3. **错误处理**：部分服务层未捕获底层异常（MinIO 连接失败冒泡到 API 层）。建议 `ServiceError` 基类统一封装。
4. **日志语义不足**：大量 `logger.info` 缺结构化字段。应改为 `logger.info("order_created", order_id=..., user_id=...)`。
5. **硬编码值**：`dispatcher.py` `.limit(50)`、`hunyuan3d/client.py` 轮询间隔应配置化。

### 3.2 前端代码质量 ⭐⭐⭐☆☆ (3/5)

**优秀实践**

1. **TypeScript 类型定义完整**：`types/design.ts`、`types/index.ts` 覆盖核心业务对象，API 响应类型齐全。
2. **组合式 API 风格统一**：全部 `<script setup>` + Composition API，composables 逻辑复用良好。
3. **状态管理清晰**：`auth.ts` Pinia setup 风格；HttpOnly Cookie + localStorage 混合策略合理（token 走 cookie，role 走 localStorage）。
4. **3D 渲染集成**：Three.js GLTFLoader 封装在 `ThreeViewer.vue`，自动旋转 + 手动拖拽逻辑清晰。

**待改进**

1. **错误处理薄弱**：`api/client.ts` 有 401 自动 refresh，但 500/网络错误缺友好提示。建议全局错误 toast + 错误边界组件。
2. **组件粒度不均**：`DesignView.vue` 300+ 行，逻辑复杂（对话+卡片+3D+派单）。建议拆分子组件。
3. **缺少加载状态管理**：多处异步请求无 loading 提示。建议统一 loading composable。
4. **硬编码文案**：提示文案直接写在组件中。建议抽离到 `constants/messages.ts` 或 i18n。
5. **样式管理**：`variables.css` 定义了主题色，但部分组件仍硬编码色值（`#f4e4d7`）。

### 3.3 数据库设计 ⭐⭐⭐⭐☆ (4/5)

**优秀设计**

1. **范式化良好**：用户/工作室/设计/订单解耦；`design_versions` 多版本（version_no + option_no）。
2. **索引策略合理**：`phone_hash`（查询）、`idempotency_key`（唯一）、`expires_at`（清理），外键索引自动创建。
3. **向量检索**：`design_templates.embedding` Vector(1536) 支持语义搜索，pgvector 集成到位。
4. **加密字段**：手机/邮箱/地址透明加密。

**待优化**

1. **时区处理**：迁移 004 添加了 `DateTime(timezone=True)`，但**未对已有数据回填时区**，老数据查询时 naive/aware 混用有风险。需执行回填脚本。
2. **缺少软删除**：订单/设计删除后无法恢复。建议添加 `deleted_at`。
3. **JSONB 字段无校验**：`design_params`/`craft_check_result` 存任意 JSON，查询需防御性编程。建议 Pydantic + JSON Schema 校验。
4. **缺少审计日志表**：敏感操作（管理员审核工作室、改订单状态）无追踪。建议新增 `audit_logs` 表。

## 四、关键功能深度分析

### 4.1 认证与授权

**实现**：JWT (HS256) + HttpOnly Cookie + Refresh Token。

**优点**：Cookie 防 XSS 窃取；role claim 编入 token 减少查库；Refresh token 机制（30 天）。

**问题**：
1. **JWT 无法主动失效**：用户改密码/权限后旧 token 仍有效。建议 Redis token 黑名单。
2. **Cookie 未设 Secure/SameSite**：`auth.py` 设置 cookie 未显式设 `secure=True, samesite="strict"`，HTTP 环境下可被截获。
3. **演示账号生产可用**：`auth.py` `DEMO_ACCOUNTS` 在生产环境应禁用，否则攻击者可用 `13800000003` 登录管理员。

### 4.2 派单算法（四维评分）

**位置**：`services/dispatch/scoring.py` + `policy.py`。

**评分维度**：工艺匹配度（50%）、负载分（20%）、评分（15%）、价格匹配度（15%）。总分 ≥ 70 分才派单，否则回退中央工作室。

**问题**：
1. **评分权重硬编码**：建议配置化（YAML 或数据库表）。
2. **无历史表现考量**：建议加入工作室历史成功率、平均交付时间。
3. **中央工作室 fallback 无验证**：`CENTRAL_STUDIO_ID` 配置错误会静默失败。建议启动时预查询验证。
4. **SQL 查询优化空间**：`dispatcher.py` 查所有 `current_load < capacity` 后在 Python 过滤。建议工艺匹配下推 SQL（`@> ANY(specialties)`）。

### 4.3 Hunyuan3D 集成

**实现**：`services/hunyuan3d/client.py` + `worker.py`。

**流程**：用户触发 → submit 调远程 API → worker 后台轮询 → 下载 GLB → 上传 MinIO → 前端轮询本地任务状态。

**优点**：任务双写（Redis + PG）保证可靠性；前后端解耦。

**问题**：
1. **轮询低效**：5 秒固定间隔。建议指数退避（5s→10s→20s→30s）。
2. **无超时保护**：`MAX_POLL_ATTEMPTS=120`（10 分钟），远程任务可能永久卡住。建议 Redis TTL + 过期清理。
3. **错误处理粗糙**：远程 500 直接标记 failed 无重试。建议 3 次指数退避重试。
4. **前端轮询无节流**：用户切走页面仍轮询。建议 `visibilitychange` 暂停。

### 4.4 文件上传（预签名 URL）

**实现**：`api/uploads.py` + `core/storage.py`。

**流程**：init 返回预签名 PUT URL → 前端直传 MinIO → confirm 后端扫描。

**优点**：预签名 URL 避免后端带宽瓶颈；状态机管理（pending → scanning → clean/quarantined）。

**问题**：
1. **ClamAV 扫描未实现**：confirm 直接标记 clean，恶意文件可访问。建议 Docker 部署 ClamAV + REST API。
2. **预签名 URL 过期时间过长**：默认 3600s。建议缩短到 300s。
3. **文件类型校验不足**：只检查 MIME type（前端可伪造）。建议后端 magic number 检测。

### 4.5 订单状态机

**位置**：`services/order/state_machine.py`。

**状态流转**：`pending → paid → accepted → in_production → fired → quality_check → shipped → delivered → completed`（旁支 cancelled/refunded）。

**优点**：状态机逻辑清晰；`OrderLog` 记录每次流转。

**问题**：
1. **缺少状态流转校验**：未强制合法转换（如 shipped 不能直接到 pending）。建议 `allowed_transitions` 字典 + 校验函数。
2. **工作室操作无权限校验**：`studio_orders.py` 更新状态时未验证订单归属，工作室 A 可改工作室 B 的订单。需 WHERE 加 `studio_id` 过滤。

## 六、性能与可扩展性

### 6.1 性能瓶颈

1. **数据库连接池偏小**：当前 `DB_POOL_SIZE=20, MAX_OVERFLOW=20`，SSE 长连接 + 多副本下不足。生产建议 `50 + 50`，监控 `pool_overflow`。
2. **N+1 查询**：`api/orders.py` 查订单列表时逐个查工作室。修复 `selectinload(Order.studio)`。
3. **Redis 单实例**：挂掉则队列/限流/缓存全挂。建议 Redis Sentinel 或 Cluster。
4. **MinIO 无 CDN**：3D 模型文件大（数 MB），直出带宽有限。建议接 CDN。

### 6.2 扩展性评估

| 维度 | 评估 | 说明 |
|---|---|---|
| 水平扩展 | ✅ 良好 | 后端无状态，可任意扩副本；worker 独立扩容 |
| 数据库扩展 | ⚠️ 中等 | 单 PG 实例 TPS 瓶颈 ~1000，建议读写分离 + Patroni |
| 对象存储扩展 | ✅ 良好 | MinIO 支持分布式模式 |
| 搜索扩展 | ⚠️ 未实现 | pgvector 单表扫描，百万级后性能下降 |

### 9.3 P0：时区数据回填（`scripts/backfill_timezone.py`）

```python
"""回填时区：naive datetime → UTC aware datetime"""
import asyncio
from sqlalchemy import text
from app.db.session import async_engine

TABLES = ['users', 'studios', 'sessions', 'session_messages',
          'design_templates', 'designs', 'design_versions', 'orders',
          'order_logs', 'studio_craft_overrides', 'idempotency_keys',
          'tasks', 'uploads']

async def backfill():
    async with async_engine.begin() as conn:
        for table in TABLES:
            await conn.execute(text(f"""
                UPDATE {table}
                SET created_at = created_at AT TIME ZONE 'UTC',
                    updated_at = updated_at AT TIME ZONE 'UTC'
                WHERE timezone('UTC', created_at) = created_at
            """))
            print(f"✓ {table} done")

if __name__ == "__main__":
    asyncio.run(backfill())
```

### 9.4 P1：订单状态机流转校验（`state_machine.py`）

```python
ALLOWED_TRANSITIONS = {
    'pending': ['paid', 'cancelled'],
    'paid': ['accepted', 'refunded'],
    'accepted': ['in_production', 'cancelled'],
    'in_production': ['fired', 'cancelled'],
    'fired': ['quality_check'],
    'quality_check': ['shipped', 'in_production'],  # 质检不过重做
    'shipped': ['delivered'],
    'delivered': ['completed'],
    'completed': [], 'cancelled': [], 'refunded': [],  # 终态
}

def validate_transition(from_status: str, to_status: str) -> bool:
    return to_status in ALLOWED_TRANSITIONS.get(from_status, [])

async def update_order_status(db, order_id, new_status, operator="system"):
    order = await get_order(db, order_id)
    if not validate_transition(order.status, new_status):
        raise ValueError(f"Invalid transition: {order.status} → {new_status}")
    old_status, order.status = order.status, new_status
    db.add(OrderLog(order_id=order_id, from_status=old_status,
                    to_status=new_status, operator=operator))
    await db.commit()
    return order
```

### 9.5 P1：工作室订单权限校验（`studio_orders.py`）

```python
@router.patch("/{order_id}")
async def update_studio_order(order_id, update, current_user=Depends(get_current_user),
                               db=Depends(get_session)):
    # 权限校验：只能操作分配给自己工作室的订单
    stmt = select(Order).where(
        Order.order_id == order_id,
        Order.studio_id == current_user.studio_id,
    )
    order = (await db.execute(stmt)).scalar_one_or_none()
    if not order:
        raise HTTPException(404, "Order not found or not assigned to your studio")
    return await update_order_status(db, order_id, update.status,
                                     operator=current_user.user_id)
```

## 十一、总结与建议

### 11.1 项目优势

1. **工程化程度高**：分层清晰、中间件完善、测试覆盖有基础。
2. **安全意识到位**：字段加密、限流、prompt 防护已实施。
3. **可观测性完备**：structlog + Prometheus + Grafana。
4. **扩展性良好**：无状态后端，水平扩展能力强。

### 11.2 当前短板

1. **生产安全漏洞**：Cookie 不安全、演示账号可用。
2. **测试覆盖不足**：派单、状态机等关键逻辑无测试。
3. **性能未验证**：无压测数据，生产容量未知。
4. **监控不完整**：缺业务指标（转化率、履约率）。

### 11.3 演进建议

**优先级**：P0（1-2 周）修复安全漏洞 → P1（3-4 周）完善监控/测试/部署 → P2（2-3 月）架构优化与功能增强。

**资源配置**：后端 2 人（安全 + 测试）、前端 1 人（错误处理 + 重构）、运维 1 人（部署 + 监控）、测试 0.5 人（性能 + e2e）。

**风险控制**：生产每日备份 + 异地容灾；灰度发布（5%→20%→100%）；每次发布前备回滚脚本；关键指标异常立即报警。

### 11.4 长期展望

ClayWords 代码质量已超过多数竞赛项目，具备商业化潜力。建议：短期（3 月）打磨 MVP 上线验证商业模式；中期（6-12 月）按用户反馈迭代、扩展工作室网络；长期（1-2 年）平台化，接入更多 AIGC 能力（图像/视频），拓展品类（木雕、刺绣）。
