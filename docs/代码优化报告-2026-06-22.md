## 四、限流系统从进程内迁到 Redis

### 旧实现的两个致命缺陷

[backend/app/core/rate_limit.py](../backend/app/core/rate_limit.py)（重写前）

1. **进程内 defaultdict** — Helm 一旦扩到多副本，限流形同虚设（每副本各自计数）
2. **死代码** — `RateLimitMiddleware` 类定义了，但 `main.py` 从未 `add_middleware`，整套限流根本没生效
3. **user 作用域永远跳过** — 中间件读 `request.state.user_id`，但项目里没有任何地方在设置它

### 新实现：Redis ZSET 滑动窗口 + Lua 原子脚本

```lua
-- 单条 Lua 完成"清过期 + 计数 + 写入"
ZREMRANGEBYSCORE key 0 (now - window)
local current = ZCARD key
if current >= limit then return {current, 0} end
ZADD key now member
PEXPIRE key window
return {current + 1, 1}
```

**关键改进**：
- ZSET score = 时间戳 → 真正的滑动窗口（不是固定窗口）
- `script_load` + `evalsha` 减少网络字节数
- **Fail-open**：Redis 不可达时返回放行而非全站 429
- user 作用域中间件**自己解析 Authorization header**，匿名访问跳过
- 在 `BaseHTTPMiddleware` 内不能 raise `HTTPException`（FastAPI 接不住），改为直接返回 `JSONResponse(429)`
- 新增 7 个单测（fakeredis + Lua 通过 lupa）覆盖：超限 / 窗口滚动 / 跨 IP 隔离 / 匿名跳过 / 路径前缀匹配 / fail-open

## 五、SSE 事件顺序漏窗口修复

### 问题

[backend/app/api/tasks.py:88-118](../backend/app/api/tasks.py#L88)（修复前）

```
1. replay history (xrange)   ← 此时 worker 推送的事件落空（pubsub 还没订阅）
2. psubscribe pubsub
3. consume live events
```

step 1 与 step 2 之间的 worker push 完全丢失。10 分钟硬超时也会砍掉长任务。

### 修复后

```
1. psubscribe pubsub          ← 先订阅，消息进 Redis 内核缓冲
2. replay history (xrange)
3. consume pubsub buffer + live, 用 event id 去重
```

**配套改动**：

[backend/app/services/tasks/task_service.py](../backend/app/services/tasks/task_service.py)

```python
# 先 xadd 拿 stream id，再 publish 携带同一 id
event_id = await redis.xadd("task:{id}:events", {...})
await redis.publish(channel, json.dumps({"id": event_id, ...}))
```

前端 SSE 用同一个 id 在 history 与 live 之间做 `seen_ids` 去重。心跳从 1s 一次降到 30s（原来每秒空发是浪费），最长生存 30 分钟（原 10 分钟会砍长 pipeline）。

## 附录 C · Token 迁移到 HttpOnly Cookie（2026-06-22 续）

承接「八、2」高价值、低风险：将 token 从 localStorage 迁移到 HttpOnly Cookie，消除 XSS 窃取风险。

### 问题

- **当前**: token 存 `localStorage`，JavaScript 可读，XSS 攻击可窃取 token 并长期冒用
- **影响面**: 50+ 处调用站点（前端 axios 拼 `Authorization` header、后端 `HTTPBearer` 依赖）

### 方案

**纯 Cookie 模式**（vs 混合模式 / 双模式）：
- token 通过 `Set-Cookie` 写入，`HttpOnly` 标志阻止 JavaScript 读取
- 浏览器自动在每个请求带 `Cookie` header（无需手动拼接）
- 前端仅持久化 `role`（用于路由守卫），不再存储 token

### 实施

#### 1. 后端改造 [app/api/auth.py](../backend/app/api/auth.py)

**移除 HTTPBearer，改为 Cookie 依赖**:
```python
from fastapi import Cookie, Response

async def get_current_user(
    access_token: str = Cookie(None),
    session: AsyncSession = Depends(get_session)
) -> UserInfo:
    if not access_token:
        raise HTTPException(401, "Missing authentication cookie")
    # ...
```

**登录/刷新端点写 Cookie**:
```python
@router.post("/login")
async def login(request: LoginRequest, response: Response, ...):
    # ... 验证逻辑
    response.set_cookie(
        key="access_token",
        value=access_token,
        httponly=True,
        secure=settings.is_production,  # 生产强制 HTTPS
        samesite="lax",
        max_age=settings.JWT_ACCESS_TOKEN_EXPIRE_MINUTES * 60
    )
    response.set_cookie(key="refresh_token", ...)
    return LoginResponse(role=user_role)  # 只返回 role
```

**新增 /logout 端点**:
```python
@router.post("/logout")
async def logout(response: Response):
    response.delete_cookie("access_token")
    response.delete_cookie("refresh_token")
    return {"message": "Logged out successfully"}
```

#### 2. 前端改造

**[stores/auth.ts](../frontend/src/stores/auth.ts)** — 移除 token 存储:
```typescript
// 仅持久化 role（用于路由守卫）
const role = ref<Role>((localStorage.getItem('role') as Role) || '')
const isAuthenticated = computed(() => !!role.value)

function setAuth(data: { role: string }) {
  role.value = data.role as Role
  localStorage.setItem('role', data.role)
}
```

**[api/client.ts](../frontend/src/api/client.ts)** — 启用 credentials + 自动 refresh:
```typescript
const client = axios.create({
  baseURL: '/',
  timeout: 30000,
  withCredentials: true  // 让浏览器自动携带 cookie
})

// 401 拦截器：自动刷新 token
client.interceptors.response.use(
  response => response,
  async error => {
    const originalRequest = error.config
    if (error.response?.status === 401 && !originalRequest._retry) {
      originalRequest._retry = true
      try {
        await axios.post('/api/v1/auth/refresh', {}, { withCredentials: true })
        return client(originalRequest)  // 重试原请求
      } catch (refreshError) {
        useAuthStore().clearAuth()
        router.push('/login')
        return Promise.reject(refreshError)
      }
    }
    // ...
  }
)
```

**其他清理**:
- [router/index.ts](../frontend/src/router/index.ts): 路由守卫改为通过 `role` 判断登录态（`!role` → 跳登录）
- [WorkOrderPopup.vue](../frontend/src/components/WorkOrderPopup.vue): `fetch(..., { credentials: 'include' })` 替代手动拼 `Authorization`
- [OrdersView.vue](../frontend/src/views/OrdersView.vue): 同上
- [types/index.ts](../frontend/src/types/index.ts): `LoginResponse` 仅保留 `role` 字段

本地无 asyncpg，无法启动完整后端，但已验证：

### 安全增强

- **XSS 防护**: `HttpOnly` 阻止 JavaScript 读取 token，XSS 无法窃取
- **CSRF 基础防护**: `SameSite=Lax` 阻止第三方站点发起跨站 POST（GET/HEAD/OPTIONS 不受限）

- **移动端适配**: 本次未覆盖，后续需求可加 `Authorization` header 回退模式
- **严格 CSRF 防护**: 可升级 `SameSite=Strict` 或加 double-submit cookie / CSRF token
