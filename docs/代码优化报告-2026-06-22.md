### 1. OrderLog `metadata=` 字段名错误 — 订单状态机直接挂

[backend/app/services/order/order_service.py:32](../backend/app/services/order/order_service.py#L32)


## 四、限流系统从进程内迁到 Redis

### 旧实现的两个致命缺陷

[backend/app/core/rate_limit.py](../backend/app/core/rate_limit.py)（重写前）

1. **进程内 defaultdict** — Helm 一旦扩到多副本，限流形同虚设（每副本各自计数）
2. **死代码** — `RateLimitMiddleware` 类定义了，但 `main.py` 从未 `add_middleware`，整套限流根本没生效
3. **user 作用域永远跳过** — 中间件读 `request.state.user_id`，但项目里没有任何地方在设置它

### 新实现：Redis ZSET 滑动窗口 + Lua 原子脚本

```lua
-- 单条 Lua 完成"清过期 + 计数 + 写入"
ZREMRANGEBYSCORE key 0 (now - window)
local current = ZCARD key
if current >= limit then return {current, 0} end
ZADD key now member
PEXPIRE key window
return {current + 1, 1}
```

**关键改进**：
- ZSET score = 时间戳 → 真正的滑动窗口（不是固定窗口）
- `script_load` + `evalsha` 减少网络字节数
- **Fail-open**：Redis 不可达时返回放行而非全站 429
- user 作用域中间件**自己解析 Authorization header**，匿名访问跳过
- 在 `BaseHTTPMiddleware` 内不能 raise `HTTPException`（FastAPI 接不住），改为直接返回 `JSONResponse(429)`
- 新增 7 个单测（fakeredis + Lua 通过 lupa）覆盖：超限 / 窗口滚动 / 跨 IP 隔离 / 匿名跳过 / 路径前缀匹配 / fail-open

## 五、SSE 事件顺序漏窗口修复

### 问题

[backend/app/api/tasks.py:88-118](../backend/app/api/tasks.py#L88)（修复前）

```
1. replay history (xrange)   ← 此时 worker 推送的事件落空（pubsub 还没订阅）
2. psubscribe pubsub
3. consume live events
```

step 1 与 step 2 之间的 worker push 完全丢失。10 分钟硬超时也会砍掉长任务。

### 修复后

```
1. psubscribe pubsub          ← 先订阅，消息进 Redis 内核缓冲
2. replay history (xrange)
3. consume pubsub buffer + live, 用 event id 去重
```

**配套改动**：

[backend/app/services/tasks/task_service.py](../backend/app/services/tasks/task_service.py)

```python
# 先 xadd 拿 stream id，再 publish 携带同一 id
event_id = await redis.xadd("task:{id}:events", {...})
await redis.publish(channel, json.dumps({"id": event_id, ...}))
```

前端 SSE 用同一个 id 在 history 与 live 之间做 `seen_ids` 去重。心跳从 1s 一次降到 30s（原来每秒空发是浪费），最长生存 30 分钟（原 10 分钟会砍长 pipeline）。

## 八、未推进的优化项

按"涉及面广、需要业务确认 trade-off"标注，留待后续 Sprint 选择。

### 高价值、低风险

1. **datetime.utcnow 升级到 timezone-aware**
   涉及 13 张表的 `DateTime → DateTime(timezone=True)` schema 改动 + alembic migration，影响所有时间字段读写。

2. **token 改 HttpOnly cookie**
   消除 XSS 窃取 access_token 的风险，但前后端联动改动较大：
   - 前端 axios 拦截器改 `withCredentials: true`，去掉 `Authorization` 头
   - 后端 `/auth/login` 写 `Set-Cookie`，`get_current_user` 改读 cookie
   - CORS 已支持 credentials，但跨域域名需校验

3. **DesignView 进一步拆分**
   建议沿 ChatPanel / PreviewCanvas / OptionCards / DispatchPanel 切分成 4 个子组件，配合本轮的 composable 后拆分成本已显著降低。

### 中价值、需要业务确认

4. **工艺匹配升级到向量相似度**
   `services/dispatch/scoring.py` 工艺匹配现在用 substring 双向 `in` 判断，"白瓷" / "薄胎白瓷" / "釉下彩白瓷" 互相误中。
   建议方案：
   - 维护 specialties 标签字典（material/category 标准化 enum）
   - 利用项目已有的 pgvector，把 `studios.specialties` 也向量化
   - `<=>` 余弦相似度替代 substring
   离线 embedding 任务需要确认 LLM 服务可用性。

5. **Order.idempotency_key 与 IdempotencyKey 表的职责梳理**
   两个机制都在用：`Order.idempotency_key` 加 `unique` 约束做 DB 层去重，`IdempotencyKey` 表存响应体用于 replay。功能不重叠但容易混淆，建议补充设计文档说明使用场景。

### 长期改进

6. **业务监控仪表盘补强**
   现有 Prometheus 指标偏基础（HTTP/DB/Redis），缺少业务维度（每小时下单数、派单成功率分布、工作室容量利用率）。

7. **CI 加测试覆盖率门槛**
   建议 `pytest --cov-fail-under=70`，避免后续 PR 降低覆盖率。
