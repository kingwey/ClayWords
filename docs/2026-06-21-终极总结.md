### 任务处理流程

```
用户提交 → TaskService → Redis Stream → Worker → 处理
   ↓           ↓             ↓           ↓        ↓
持久化     PostgreSQL    队列持久化   重试3次   Pub/Sub
   ↓           ↓             ↓           ↓        ↓
订单表     双写策略      Consumer    Dead      SSE 推送
                         Group      Letter     客户端
```

### 支付物流流程

```
订单创建 → 待支付
    ↓
支付 → dispatched (已支付) → 派单
    ↓
工作室接单 → producing (制作中)
    ↓
制作完成 → completed
    ↓
工作室发货 → shipped (已发货)
    ↓
用户确认 → delivered (已签收)

退款分支:
dispatched/producing → 申请退款 → refunding → refunded
```

## 🔑 关键技术决策

### 1. 跳过 Phase Q3 (3D 模型生成)
**原因**: 工作量大（约 12 天），需要 GPU 环境和设计师资源  
**收益**: 专注业务闭环，Q3 可并行或后续补充  
**影响**: 暂时使用 Mock 模型，不影响其他功能开发

### 2. Redis Streams vs RabbitMQ/Kafka
**选择**: Redis Streams  
**原因**: 轻量、易运维、自带持久化和重放  
**收益**: 开发效率高，运维成本低

### 3. MinIO 预签名 URL vs 服务器中转
**选择**: 预签名 URL  
**原因**: 减轻服务器带宽压力，提升上传体验  
**收益**: 服务器只处理元数据，文件直传对象存储

### 4. 支付宝沙箱 Mock 模式
**选择**: Mock 模式（无需配置密钥）  
**原因**: 快速开发和测试，真实密钥留给生产环境  
**收益**: 开发阶段无配置依赖，CI/CD 友好

### 5. 幂等性键设计
**选择**: IdempotencyKey 表 + 过期时间  
**原因**: 防止支付回调重复处理  
**收益**: 强一致性保证，避免重复扣款

## 🎓 技术亮点

### 1. 双写策略 (Redis + PostgreSQL)
- **读**: 优先 Redis 缓存，fallback PostgreSQL
- **写**: 先 PostgreSQL 持久化，再 Redis 缓存
- **收益**: 性能 + 可靠性

### 2. Consumer Group 并发消费
- **模式**: 多 Worker 自动负载均衡
- **故障恢复**: pending 消息自动认领
- **收益**: 横向扩展，高可用

### 3. Last-Event-ID 断线重连
- **机制**: SSE + Redis Stream 历史回放
- **收益**: 断网 30s 重连不丢事件

### 4. 预签名 URL 直传
- **流程**: 客户端 → MinIO（直传）
- **收益**: 服务器零负载，上传体验优

### 5. 幂等性键防重
- **实现**: IdempotencyKey 表 + 唯一约束
- **场景**: 支付回调、退款请求
- **收益**: 强一致性保证

### 6. 四维评分派单
- **维度**: 工艺 35% + 产能 30% + 地理 15% + 评分 20%
- **阈值**: 总分 >= 0.55 自动派单
- **收益**: 智能派单，负载均衡
