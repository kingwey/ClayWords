## 一、优先级判断分析

### 1.1 应提升至 P0 的项目

| 序号 | 项目 | 原分类 | 建议分类 | 理由 |
|------|------|--------|----------|------|
| 1 | **DB 会话未及时释放** | P1 | **P0** | SSE 长连接持有 DB session + 连接池只有 20 → 20 个并发 SSE 连接就耗尽连接池 → **生产事故** |
| 2 | **HTTP 客户端无连接池** | P1 | **P0** | 每次调用混元3D/LLM 都创建新 TCP 连接，高并发下会触发系统 `Too many open files` 错误 |

**验证依据**:
```python
# backend/app/api/sse.py - SSE 端点注入了 DB session
@router.get("/task/{task_id}/progress")
async def stream_task_progress(
    task_id: str,
    session: AsyncSession = Depends(get_session),  # ⚠️ 长连接持有会话
):
    async for event in _stream_events(task_id):
        yield event
    # session 在连接关闭前一直占用
```

### 1.2 可降级至 P1 的项目

| 序号 | 项目 | 原分类 | 建议分类 | 理由 |
|------|------|--------|----------|------|
| 1 | **JWT 黑名单内存问题** | P0 | P1 | 假设日活 1万用户，每天登出 10%，每个 token 1KB → 1MB/天，Redis 内存压力很小 |
| 2 | **MinIO URL 暴露** | P0 | P1 | 预签名 URL 已设置 1 小时过期 + MinIO 可配置 Bucket Policy 限制来源 IP，风险可控 |

## 二、遗漏的关键问题

### 2.1 支付回调竞态条件 (P0)

**问题描述**:  
支付宝回调可能重复发送（网络重试），当前幂等性检查存在竞态窗口：

```python
# backend/app/api/payments.py:162-170
# 步骤 1: 查询幂等性键
stmt = select(IdempotencyKey).where(IdempotencyKey.key == idempotency_key)
existing = result.scalar_one_or_none()
if existing:
    return "success"  # 已处理

# 步骤 2: 写入幂等性键
session.add(idem_key)
# ⚠️ 竞态窗口：两个并发回调都通过了步骤 1，都会执行步骤 2
```

**影响**: 订单重复扣款、工作室容量重复占用、支付记录重复。

**优化方案**:
```python
# 方案 1: 数据库唯一约束 + 异常捕获
try:
    session.add(idem_key)
    await session.flush()  # 强制写入，触发唯一约束检查
except IntegrityError:
    await session.rollback()
    return "success"  # 重复回调，已处理

# 方案 2: Redis 原子 SETNX
if not await redis_client.setnx(f"payment_lock:{trade_no}", "1", ex=300):
    return "success"  # 已被其他回调处理
```

### 2.2 工作室容量释放缺失 (P0)

**问题描述**:  
订单取消/退款时，未释放工作室 `current_load`，导致容量泄漏。

```python
# backend/app/services/order/order_service.py:93-127
async def cancel_order(...):
    # ⚠️ 只检查了状态，没有释放容量
    if order.studio_id:
        # 应该: await release_studio_capacity(db, order.studio_id)
        pass
```

**影响**: 工作室容量逐渐耗尽，无法接新单。

**优化方案**:
```python
async def cancel_order(db, order_id, ...):
    order = await _get_order(db, order_id)
    
    # 释放工作室容量
    if order.studio_id and order.status in ["dispatched", "producing", ...]:
        await db.execute(
            update(Studio)
            .where(Studio.studio_id == order.studio_id)
            .values(current_load=Studio.current_load - 1)
        )
        
        # 更新 Prometheus 指标
        from app.core.metrics import metrics
        studio = await db.get(Studio, order.studio_id)
        metrics.set_studio_load(studio.studio_id, studio.current_load - 1)
```

### 2.3 Redis Pub/Sub 消息丢失 (P1)

**问题描述**:  
SSE 使用 Redis Pub/Sub 推送进度，但 Pub/Sub 无持久化：
- 客户端断线重连时，中间的消息已丢失
- 无法实现断点续传

**影响**: 用户刷新页面后看不到任务进度，体验差。

**优化方案**:
```python
# 方案 1: Pub/Sub + Streams 混合
# 1. 进度写入 Redis Streams (有序、可回溯)
await redis_client.xadd(f"task:{task_id}:progress", {"stage": "gen", "percent": 60})

# 2. 同时 Pub 到 Pub/Sub (实时推送)
await redis_client.publish(f"task:{task_id}", json.dumps({"stage": "gen", "percent": 60}))

# 3. SSE 连接时先回放 Streams 历史，再订阅 Pub/Sub
async def stream_task_progress(task_id: str, last_event_id: str = None):
    # 回放历史
    if last_event_id:
        entries = await redis_client.xrange(f"task:{task_id}:progress", min=last_event_id)
        for entry_id, data in entries:
            yield f"id: {entry_id}\ndata: {json.dumps(data)}\n\n"
    
    # 订阅实时更新
    async for message in redis_client.subscribe(f"task:{task_id}"):
        yield f"data: {message}\n\n"
```

### 2.4 ClamAV 扫描阻塞上传流程 (P1)

**问题描述**:  
当前上传流程是同步扫描：

```
1. 前端上传到 MinIO (直传)
2. POST /uploads/{id}/confirm
3. 后端从 MinIO 下载文件
4. 调用 ClamAV 扫描 (阻塞 10-30 秒)  ⚠️
5. 返回结果
```

**影响**: 用户体验差，上传接口响应慢。

**优化方案**:
```python
@router.post("/uploads/{upload_id}/confirm")
async def confirm_upload(upload_id: str, ...):
    # 立即返回 202 Accepted，异步扫描
    upload.state = "scanning"
    await session.commit()
    
    # 提交到 Worker 队列异步扫描
    await arq_client.enqueue_job(
        "scan_upload",
        upload_id=upload_id,
        object_key=upload.object_key
    )
    
    return {"upload_id": upload_id, "state": "scanning"}

# 前端轮询 GET /uploads/{id} 获取扫描结果
```

### 2.5 Worker 无任务超时机制 (P1)

**问题描述**:  
Worker 任务可能因外部 API 挂起、网络故障等原因永久卡住，占用 worker 槽位。

```python
# worker/main.py:42-168
async def process_design_gen(ctx, session_id, design_params, task_id):
    # ⚠️ 无全局超时控制
    # 如果 hunyuan3d API 挂起，任务永久占用 worker
    result = await call_hunyuan3d_api(...)
```

**影响**: Worker 池耗尽，新任务无法调度。

**优化方案**:
```python
import asyncio

async def process_design_gen(ctx, session_id, design_params, task_id):
    try:
        # 设置全局超时 5 分钟
        async with asyncio.timeout(300):
            await _do_generation(session_id, design_params, task_id)
    except asyncio.TimeoutError:
        # 标记任务失败
        await redis_client.set(
            f"task:{task_id}:result",
            json.dumps({"state": "failed", "error": "任务超时"}),
            ex=3600
        )
        raise
```

## 三、代码示例问题

### 3.1 httpx 客户端缺少清理逻辑

**原示例问题**:
```python
class HTTPClientManager:
    _client: httpx.AsyncClient = None
    
    @classmethod
    async def get_client(cls) -> httpx.AsyncClient:
        if cls._client is None or cls._client.is_closed:
            cls._client = httpx.AsyncClient(...)
        return cls._client
    # ⚠️ 缺少 close() 方法
```

**修复**:
```python
class HTTPClientManager:
    _client: httpx.AsyncClient = None
    
    @classmethod
    async def get_client(cls) -> httpx.AsyncClient:
        if cls._client is None or cls._client.is_closed:
            cls._client = httpx.AsyncClient(
                timeout=httpx.Timeout(30.0, connect=10.0),
                limits=httpx.Limits(max_connections=100, max_keepalive_connections=20),
            )
        return cls._client
    
    @classmethod
    async def close(cls):
        """应用关闭时调用"""
        if cls._client and not cls._client.is_closed:
            await cls._client.aclose()
            cls._client = None

# backend/app/main.py - 注册 shutdown 事件
@app.on_event("shutdown")
async def shutdown():
    await HTTPClientManager.close()
```

### 3.2 SSE 扇出示例缺少订阅者清理

**原示例问题**:
```python
class SSEManager:
    def __init__(self):
        self._subscribers: dict[str, set[asyncio.Queue]] = defaultdict(set)
    
    async def subscribe(self, task_id: str, user_id: str) -> asyncio.Queue:
        queue = asyncio.Queue(maxsize=100)
        self._subscribers[task_id].add(queue)
        return queue
    # ⚠️ 客户端断开后，queue 没有从 _subscribers 移除
```

**修复**:
```python
class SSEManager:
    async def subscribe(self, task_id: str, user_id: str) -> asyncio.Queue:
        await self._verify_task_ownership(task_id, user_id)
        
        queue = asyncio.Queue(maxsize=100)
        self._subscribers[task_id].add(queue)
        
        # 返回带清理逻辑的上下文管理器
        return _SubscriptionContext(self, task_id, queue)

class _SubscriptionContext:
    def __init__(self, manager: SSEManager, task_id: str, queue: asyncio.Queue):
        self.manager = manager
        self.task_id = task_id
        self.queue = queue
    
    async def __aenter__(self):
        return self.queue
    
    async def __aexit__(self, *args):
        # 客户端断开，清理订阅
        self.manager._subscribers[self.task_id].discard(self.queue)
        if not self.manager._subscribers[self.task_id]:
            # 该任务无订阅者，取消 Redis 订阅
            await self.manager._stop_redis_subscription(self.task_id)

# 使用
async with sse_manager.subscribe(task_id, user_id) as queue:
    while True:
        event = await queue.get()
        yield event
```

### 3.3 工作室评分缓存缺少失效机制

**原示例问题**:
```python
# 缓存 30 秒，但工作室信息变更时没有主动失效
await redis_client.set(cache_key, json.dumps(studios), ex=30)
```

**修复**:
```python
# 1. 定义缓存键生成规则
def get_studios_cache_key() -> str:
    return "studios:available:v1"

# 2. 工作室信息变更时主动失效
async def update_studio(db, studio_id, updates):
    await db.execute(
        update(Studio).where(Studio.studio_id == studio_id).values(**updates)
    )
    await db.commit()
    
    # 失效缓存
    await redis_client.delete(get_studios_cache_key())

# 3. 容量占位/释放时也要失效
async def dispatch_to_studio(...):
    # ... CAS 占位成功后
    await redis_client.delete(get_studios_cache_key())
```
